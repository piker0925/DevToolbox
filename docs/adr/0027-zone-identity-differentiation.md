# ADR-0027 구역 아이덴티티 차별화 — 공통 골격 + 구역별 톤

## 상태

확정 (2026-07-17) — 046(구역 인프라) 구현 완료 직후 결정. 적용은 045(랜딩)부터, 구역 홈 히어로 보강은 후속 이슈.

## 배경

046까지의 구역 구현은 DESIGN.md §2-1의 "구역 액센트는 컨텍스트 표시용" 원칙을 따랐다 — 구역 간 차이는 액센트 컬러 하나뿐이고 레이아웃·밀도·톤은 완전히 동일하다. 사용자 검토에서 이 수준이 의도와 다르다는 피드백이 나왔다: 목표는 네이버의 웹툰/지도/뉴스처럼 **한 브랜드 안에서 구역마다 뚜렷이 다른 서브 프로덕트 느낌**을 주는 것이다.

이는 미적 취향이 아니라 구조적 필연이다. 3차에서 `/fun`에 미니게임 8종, `/life`에 생활 계산기가 들어오는데, 게임 구역이 SQL 포맷터와 같은 워크벤치 톤이면 그 자체가 UX 실패다. ADR-0026이 이미 `kind`별 셸(GamePage 등)을 예정해뒀으므로 구역별 톤 차이는 어차피 발생한다 — 이 ADR은 그것을 우연이 아니라 의도된 설계로 만든다.

## 결정

**공통 골격은 통일하고, 구역별 아이덴티티는 세 가지 수단으로만 차별화한다.**

### 통일 (불변 — 건드리면 통일성이 깨진다)

- 타이포 스케일·웨이트 체계 (§3)
- radius·spacing 스케일 (§4)
- 컴포넌트 형태: 카드·버튼·인풋·사이드바 구조 (§6)
- 인터랙션 패턴: hover/focus 트랜지션, 스태거, reduced-motion (§4)

### 차별화 (구역별로 다르게 — 세 수단으로 한정)

1. **토큰**: 구역 액센트(`--zone-accent`, 046에서 `data-zone` 전환 인프라 완성) + 파생 틴트(히어로 배경 등). 원시 hex/oklch 직접 사용 금지는 유지 — 전부 style.css 토큰을 거친다.
2. **데이터**: `zones.ts` 구역 메타에 히어로 카피·아이콘 무드 등 아이덴티티 필드 추가. 렌더는 단일 `ZoneHomePage`가 데이터로 분기 — 구역별 컴포넌트 포크 금지.
3. **kind별 셸** (ADR-0026): 게임은 GamePage, 뷰어는 ViewerPage — 구역 톤의 상당 부분은 콘텐츠 셸에서 자연스럽게 나온다.

### 구역별 톤 가이드

| 구역 | 톤 | 예 |
|------|-----|-----|
| `/dev` | 기술적·밀도 높음 | 모노스페이스 적극 활용, 간결한 카피 |
| `/files` | 차분·기능적 | 문서 작업실 느낌, 절차 중심 카피 |
| `/life` | 따뜻함·친근함 | 부드러운 카피 톤, 생활어 |
| `/fun` | 경쾌함·놀이 | 과감한 액센트 활용, 짧고 가벼운 카피 |

### 금지 (오버엔지니어링 경계)

- 구역마다 별도 레이아웃 엔진/컴포넌트 트리 — 유지보수 비용만 커진다
- 구역마다 커스텀 일러스트 제작 — 리소스 없이 어색해짐, 아이콘 조합까지만
- 구역별 폰트 도입 — §3의 "새 폰트 추가 금지" 유지

## 적용 순서

1. 045(랜딩): 구역 카드 4장이 각 구역 아이덴티티를 처음 표현하는 자리 — 여기서 4구역의 시각 언어 확정
2. 구역 홈 히어로 보강(후속 소이슈): 045에서 확정된 언어를 구역 홈 상단에 역적용
3. 3차 기능 이슈들: kind별 셸 구현 시 이 톤 가이드를 따름

## 영향

- DESIGN.md §2-1 개정: "구역 액센트는 컨텍스트 표시용" → "구역 액센트 + 구역 히어로 톤" 확장. §5의 "마케팅 문법은 대문 한정" 예외에 구역 홈 히어로 영역을 제한적으로 추가.
- ADR-0023(포털 IA)·ADR-0026(카탈로그)과 충돌 없음 — 오히려 0026의 "데이터로 분기, 컴포넌트 포크 금지" 철학을 구역 레벨로 확장한 것.
